Qwen3-4B 电商生图工具路由微调

我把“用户一句自然语言需求”拆成小模型可学习的工具调用任务：选择正确工具、抽取关键参数、输出合法 JSON，并在信息不足时主动追问。这个案例完整跑通了数据构建、人工抽检、QLoRA 训练、评测报告和 LoRA 产物交付。

业务问题

电商生图平台里，用户不会按工具名下指令，而是说“帮我做一组图”“把背景换成厨房”“这张图合不合规”。如果所有请求都交给大模型自由发挥，成本高、输出不稳定、也难接后端执行。这里的目标是训练一个小模型路由器，把自然语言稳定转成平台内部可执行的工具调用 JSON。

这个模型不是为了直接生成图片，而是作为 Agent 前置路由层：先判断该调用哪个工具，再把参数结构化，后续才交给图像生成、编辑、合规或分析工具执行。

完整流程

1. 目标定义把电商生图需求定义为工具路由和 JSON 参数抽取问题。
2. Schema 设计整理 19 个工具、固定字段、枚举值和追问规则。
3. 数据构建生成 3600 条样本，并做人审抽检和去重清洗。
4. 训练切分3000 训练、450 常规评测、150 困难评测。
5. QLoRA 训练在 A100 上用 Unsloth 跑 Qwen3-4B LoRA。
6. 效果验证输出逐条 CSV、汇总报告和错误类型分布。

我负责的拆解

  • 定义平台工具白名单：从 9 个基础工具扩展到 19 个工具，覆盖生成、编辑、合规、套图规划、文案、参考图分析等场景。
  • 把“模糊需求”单独设计成 `ask_followup`，避免模型在关键信息缺失时硬编参数。
  • 把“闲聊/非电商任务”设计成 `direct_text_response`，避免所有输入都被误路由到图像工具。
  • 对 Qwen3 输出中的思考模板干扰做了处理，训练样本只保留 assistant 的合法 JSON。

训练配置

  • Base model：Qwen3-4B 4bit 版本。
  • 方法：Unsloth + QLoRA / SFT。
  • GPU：A100 40GB。
  • Epoch：3；Total steps：1125。
  • LoRA rank：16；LoRA alpha：16。
  • 最终训练 loss：0.1139。
  • 可训练参数约 0.81%，用于证明这是轻量 LoRA 微调而不是全量重训。

效果验证

98.67%工具选择准确率
83.50%可执行级结果，综合分不低于 1
1.22 / 2平均参数得分

评测口径不是只看“像不像”，而是拆成 JSON 是否可解析、工具是否正确、参数是否可执行、错误类型是什么。这样能把模型效果转成后续可迭代的数据问题。

工具调用样例

用户原话

把这张玩具车的背景去掉，要透明底，边缘 refine

{"tool":"remove_background","arguments":{"image_reference":"latest","output_background":"transparent","edge_refine":true},"need_user_confirmation":false}

用户原话

规划一套亚马逊 Listing 的图片，包括主图和六张附图

{"tool":"plan_image_set","arguments":{"product_type":"electronics","image_set_type":"listing_set","platform":"amazon","image_count":7,"key_messages":["features","size","usage"]},"need_user_confirmation":false}

用户原话

需要一款儿童玩具在京东主图的画面构思，要温馨一点

{"tool":"generate_creative_brief","arguments":{"product_type":"儿童玩具","goal":"main_image","style_direction":"温馨","scene_direction":"unknown","platform":"jd","output_count":1},"need_user_confirmation":false}

交付物

LoRA adapter：包含 adapter 权重、adapter config、tokenizer 和 chat template。
评测报告：包含总览、按 split、按工具和错误类型分布。
逐条评测 CSV：保留输入、期望工具、预测工具、解析结果和错误类型。
数据集报告：记录 3600 条数据的切分和 19 个工具分布。

复盘结论

  • 工具选择已经比较稳定，主要问题集中在参数精确度和复杂工具字段覆盖。
  • 仅看训练 loss 不够，必须配套结构化评测，否则无法判断模型是否真的可执行。
  • 下一轮应该优先补 `plan_image_set`、`generate_creative_brief`、`generate_marketing_copy` 等参数较复杂工具的困难样本。

这个案例展示的是一个可复用的小模型 Agent 路由闭环：业务任务定义、数据 schema、训练、评测、bad-case 回流，而不是单次调参。
